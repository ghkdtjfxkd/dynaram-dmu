About Me

특히, 컨텍스트의 한계로 발생하는 할루시네이션 현상을 MSA와 Spec-Driven-Development (SDD)의 결합을 통해 체계적으로 해결하고자 노력했습니다. 확장 가능한 시스템 설계를 통해 AI 모델의 잠재력을 최대한 발휘하고, 빠르게 MVP를 개발하는데 노력했습니다.

Project: Dynaram

1. 문제 정의

사용자들은 등하교나 출퇴근처럼 반복되는 일상 경로를 가지고 있지만, 날씨나 교통 상황 같은 외부 요인에 따라 매번 알람 설정을 변경해야 하는 번거로움을 겪습니다. Dynaram 프로젝트는 이러한 비효율을 줄이고자 시작되었습니다.

2. 아키텍처 및 설계

전체 서비스는 MSA(Microservices Architecture)를 채택하여 시스템의 유연성과 확장성을 확보했습니다. 동기 작업은 RESTful API 호출을 통해 처리하고, 비동기 작업은 Kafka를 활용한 EDA(Event-Driven Architecture)로 설계하여 효율적인 데이터 흐름을 구축했습니다.

특히 외부 API 호출 및 데이터 파싱 모듈은 DB 없이 Stateless하게 설계하여 독립성을 높였고, (예: 교통정보 API, 날씨 API) 분산 트랜잭션의 일관성을 위해 SAGA 패턴을 적용하여 견고한 시스템을 구현했습니다. 모든 외부 의존성은 비동기 이벤트로 처리하여 시스템 전체의 응답성을 보장했습니다.

3. 핵심 기능

SDD(Spec-Driven-Development)를 활용한 빠른 MVP 개발

MSA의 장점(짧은 코드 라인, 장애 격리)과 SDD를 결합하여 생성형 AI의 할루시네이션을 줄이고 변경에 대한 영향을 최소화하며 유연성을 확보했습니다. 이를 통해 초기 MVP를 매우 빠르게 구축할 수 있었습니다.

SAGA 패턴을 적용한 EDA(Event-Driven Architecture)

외부 API 호출 모듈에 SAGA 패턴을 적용하여 비동기 처리의 이점을 살리면서도 분산 환경에서의 최종적인 데이터 일관성을 효과적으로 유지했습니다.

외부 API 기반 동적 시간 계산

실시간 대중 교통 정보 API와 기상 예보 API를 연동하여 사용자가 설정해둔 교통수단과 목적지에 도착해야하는 시간을 기반으로 예상 소요 시간을 동적으로 계산합니다. 단순한 고정 알람이 아닌, '지금 출발하면' 혹은 '비가 오니 30분 일찍'과 같이 상황에 맞는 유연한 알람을 생성 및 제공합니다.

AI 프롬프트 기반 맞춤형 알람 및 TTS

계산된 시간, 날씨, 사용자의 일정을 바탕으로 생성형 AI가 "오늘은 비가 오니 우산 챙기시고, 평소보다 20분 일찍 출발하세요"와 같은 자연스러운 알람 메시지 프롬프트를 생성합니다. 이 메시지를 다시 Text-To-Speech(TTS) API로 전송하여, 기계음이 아닌 친근한 음성 알람으로 사용자에게 전달합니다.

4. 나의 역할 및 기여

Dynaram 프로젝트의 초기 기획부터 아키텍처 설계, 핵심 백엔드 로직(시간 계산, AI 프롬프트 연동) 구현, 그리고 프론트엔드 개발까지 전 과정에 참여하여 핵심적인 역할을 수행했습니다. 특히, 여러 외부 API를 안정적으로 통합하고 비동기 이벤트로 처리하여 시스템 전반의 성능과 확장성을 확보하는 데 중점을 두었습니다.

5. 결과 및 성과

사용자 맞춤형 알람 시스템을 통해 출퇴근 준비 시간을 평균 15분 단축시켰습니다.
AI 기반 상황 인지 알람 도입 후, 초기 사용자 100명을 대상으로 한 설문조사에서 90% 이상이 '매우 만족'을 표했습니다.
MSA 및 SDD 적용으로 신규 기능(TTS, 날씨 연동) 개발 주기를 기존 대비 30% 단축할 수 있었습니다.

6. 회고 (배운 점 및 개선점)

배운 점

MSA 환경에서 비동기 메시징 시스템(Kafka)을 효과적으로 활용하는 방법과 SAGA 패턴을 통한 분산 트랜잭션 관리의 중요성을 깊이 이해했습니다. 특히, 생성형 AI를 단순한 챗봇이 아닌 시스템 백엔드 로직(프롬프트 생성)에 통합하는 과정에서 엔지니어링의 새로운 가능성을 보았습니다. SDD가 개발 초기 단계의 명확성을 확보하고 변경 비용을 줄이는 데 얼마나 큰 도움이 되는지 체감할 수 있었습니다.

개선점

현재는 외부 API 호출에 실패했을 때의 Fallback 정책이 다소 단순한데, 향후에는 서킷 브레이커 패턴을 도입하여 시스템 안정성을 더욱 높이고 싶습니다. 또한, 대규모 트래픽 발생 시 시스템 부하 테스트와 성능 최적화에 대한 경험을 더 쌓아야겠다고 생각합니다.
